here is my grand vision, i'm calling it amoebius. it is a haskell binary that runs as a cli tool, a host level daemon with sudo powers, and also a daemon running in a k8s node. it treats multiple substrates as first class citizens: apple silicon, windows, linux (with optional cuda support). once the binary is built, it takes responsibility for the remainder of the boostrap. it is k8s centric. it supports kind, rke2 (HA mode), and AKS as first class citizens. the binary has a bootstrap command where it is given a bootstrap.dhall with a) instructions for which k8s substrate will be used, and b) any credentials needed to deploy them (eg ssh keys for k8s nodes, aws credentials). the k8s cluster is always deployed with the same core services: harbor (all containers are cached locally and pulled from harbor, once harbor is up and responsive), MinIO, hashicorp vault, pulsar, prometheus/grafana, keycloak. there is always a LoadBalancer available in k8s-- either a cloud native LB, or MetalLB. All services run on an HA basis. whenever we use postgres, it must be patroni backed via the percona plugin. each service requiring postgres gets a separate deployment, with its own pgadmin. keycloak manages all ingress from the wild (including localhost connections) through the LB via the k8s gateway api. the only exception to the LB->Keycloak rule is that the amoebius binary on the host can a) directly talk to the kubeapi via the default mTLS created by the k8s distro, and b) the host binary can deploy other host level services outside of the k8s cluster which can talk to the cluster's services via NodePort. (k8s deployments are generally preferred, but sometimes host level services are needed for platform reasons, eg an inference engine that wants to use apple silicon's unified memory) these access points are strictly available from the localhost, there is to be no WAN or LAN access.

## Cluster Manifest

A cluster manifest is a .dhall describing one or more amoebius apps, as well as general cluster settings. it specifies things like a) PKI trust anchor (self or parent), b) vault unlock model (self-unlock or parent unlock), c) 

It also makes it impossible to represent illegal states like: a) apps with redundant names b) insecure ingress (all traffic must be keycloak authorized) c) non-compliant durable storage models d) 

## Amoebius Apps

An amoebius app consists of two things a) one or more (potentially publically available) container images / Dockerfile that builds correctly for both amd64 and arm64 architectures b) an app spec .dhall file satisfying certain criteria, including what services (if any) are LB services, the rules for keycloaked backed auth, any durable storage required (MinIO, block storage, postgres), and pulsar topic lifecycles.

There is a generic & reusable helm chart for amoebius apps. This does not have to be provided explicitly, the values will be populated automatically and managed dynamically (eg dynamic replicas). Amoebius translates the app requirements into services, statefulsets, etc 

In genearal, each app a) must have a name unique for that cluster b) is deployed in its own namespace with the same name c) may create one or more MinIO buckets named according to <app_name>/<bucket_name> d) may provision block storage using the kubernetes.io/no-provisioner storage policy e) may request a postgres DB that will live in the same namespace.
