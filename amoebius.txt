here is my grand vision, i'm calling it amoebius. it is a haskell binary that runs as a cli tool, a host level daemon with sudo powers, and also a daemon running in a k8s node. it treats multiple substrates as first class citizens: apple silicon, windows, linux (with optional cuda support). once the binary is built, it takes responsibility for the remainder of the boostrap. it is k8s centric. it supports kind, rke2 (HA mode), and AKS as first class citizens. the binary has a bootstrap command where it is given a bootstrap.dhall with a) instructions for which k8s substrate will be used, and b) any credentials needed to deploy them (eg ssh keys for k8s nodes, aws credentials). the k8s cluster is always deployed with the same core services: harbor (all containers are cached locally and pulled from harbor, once harbor is up and responsive), MinIO, hashicorp vault, pulsar, prometheus/grafana, keycloak. there is always a LoadBalancer available in k8s-- either a cloud native LB, or MetalLB. All services run on an HA basis. whenever we use postgres, it must be patroni backed via the percona plugin. each service requiring postgres gets a separate deployment, with its own pgadmin. keycloak manages all ingress from the wild (including localhost connections) through the LB via the k8s gateway api. the only exception to the LB->Keycloak rule is that the amoebius binary on the host can a) directly talk to the kubeapi via the default mTLS created by the k8s distro, and b) the host binary can deploy other host level services outside of the k8s cluster which can talk to the cluster's services via NodePort. (k8s deployments are generally preferred, but sometimes host level services are needed for platform reasons, eg an inference engine that wants to use apple silicon's unified memory) these access points are strictly available from the localhost, there is to be no WAN or LAN access. after bootstrap, the binary has daemons for host and cluster, in both cases it is given a declarative .dhall.
