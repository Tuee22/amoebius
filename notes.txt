- The parent-node chicken-and-egg, resolved. The tension: an InForceSpec describes a cluster (nodes, secrets), but it is uploaded THROUGH a singleton that already runs INSIDE a cluster -- so how does the first/parent node know what to spin up with no parent InForceSpec above it? The cycle breaks by SEPARATING SUBSTRATE GENESIS FROM SPEC RECONCILIATION, and grounding the recursion in the operator:
    - Genesis is a FrameConfig concern, not an InForceSpec. Every node first comes up as the zero-secret single-node root -- a degenerate corner of the typed topology selected by `bootstrap --distro` + the transient binary-sibling amoebius.dhall, just enough to run the platform floor + the singleton + admin REST. No spec is required to reach it, so there is nothing to receive before the receiver exists.
    - The InForceSpec is the reconcile TARGET, supplied from outside and above. It only enters once the singleton is up, addressed to it. A node never self-authors its own description: it is handed one -- by the operator for the root, by the parent for a child.
    - The recursion bottoms out at the operator. Doctrine's own framing: "the first cluster is bootstrapped on a host; every later cluster is spawned by a parent -- both end in the same place." - InForceSpec migrations, and why data-destruction / cross-tenant-sharing are unrepresentable. An "InForceSpec migration" is just the typed diff between the in-force RootInForceSpec and a newly-uploaded one, delivered through `dhall update` and realized by the ordinary reconcile (desired = render(spec); the diff is typed). It is NOT a new operation, and a DB schema migration is only ONE RolloutPhase inside it. Two guarantees ride on top:
    - No destruction of user data. Type-foreclosed first: the only verbs a migration may author over an existing durable coordinate live in a closed union StorageMutation = Retain RetentionPolicy | Offload target | Grow size | Shrink size, where Shrink is DEFINED as create-new -> verified-migrate -> retire-old, never in-place truncation. There is no Delete/Drop/Erase/Truncate arm anywhere the spec can reach, so "destroy these bytes" has no inhabitant. Decode-foreclosed second (the genuinely new fold, since Dhall can't compare two whole specs at the type level): the old->new diff is rejected at `dhall update` if it would ORPHAN a retained coordinate (dropped from the active spec with no data-preserving disposition) or SHRINK retention below the currently-retained span without an offload-first phase. A rename is a move over a FIXED identity (a PV/bucket/topic identity is a total function of (tenant, app, name)), so it can never degrade to a delete-then-create that loses data.
    - Test teardown is allowed WITHOUT opening a production-destroy hole, because teardown lives on a DIFFERENT TYPE. A test topology's coordinates are TestOwned (they carry the mandatory teardown obligation and are the only shape the elevated flagged-delete credential can name); production data is DataResidence t a and has no Drop arm at all. No constructor converts one to the other, so the destroy capability is structurally withheld from anything holding user data.
    - No cross-tenant sharing. Every durable datum is a phantom-tenant-tagged Ref t a with a single-owner index (tenant, app, name). There is no retag :: Ref t1 a -> Ref t2 a, no reassignOwner, no mergeTenants -- a datum's tenant-owner is an immutable coordinate. Reinforced cryptographically by the per-tenant Vault Transit key (a tenant can't even decrypt another's envelope) + Pulsar namespace + MinIO bucket policy, and belt-and-suspenders by an owner-immutability diff fold (every surviving coordinate's (tenant,app) owner must be byte-identical old->new, catching "steal by re-declaration").
  Migration RolloutPhases (ordered, readiness-gated, expand->migrate->contract so nothing destructive fires before verification): DB schema = create-new/verified-migrate/retire-old; topic retention-shrink = offload-before-shrink; PV shrink = new-volume/copy/verify-before-retire; rename = dual-name/copy-verify/drop-alias; reader cutover = Pulsar subscription or Gateway-API weight shift. Rollback is free: the prior spec is still an immutable Release in the ledger, so rollback = CAS the environment pointer back to the prior releaseHash and let the reconciler converge (it finds data intact because no phase destroyed anything unverified); PromotionGate still forecloses unverified->prod. Honest limit: a type can't prove a live box still has its disk or that a verified copy's bytes match -- the actual reclaim of a retired volume is an elevated, flagged, runtime-checked action, and the orphan check may need to consult live retained-state (a hybrid decode+runtime gate, labelled honestly). Open: keep the destructive Retire verb OUT of the spec (elevated action) vs a privileged auditable spec arm; and whether sanctioned sharing (app B continues app A's model with a grant) becomes an append-only, revocable capability EDGE rather than a re-tag.   CLIENT REDIRECTION during a migration -- how a live session rebinds to the new gateway (and why "just repoint DNS and hope" is insufficient). DNS TTL + resolver caching leave a window where clients still resolve the OLD IP; a mid-session client gets STUCK if the old gateway is hard-frozen at the INGRESS. Handling, so a session can ALWAYS rebind:
      - Freeze WRITES (the consistency boundary), NOT the ingress. Keep the OLD gateway alive as a TRANSPARENT REVERSE PROXY to the new (A -> B over the fabric) for the whole DNS-drain window -- a client still resolving A is silently forwarded to B (its session state is on B). Primary mechanism: seamless, NO client-visible host change, NO cookie-scoping dependency.
      - Steady-state LOW TTL on the migrating "active" record (bounds the split window; TTL can't be shortened retroactively, so it must already be low). Each cluster ALSO has a STABLE per-cluster address (name/IP that never migrates) -- the proxy target and the explicit-redirect fallback.
      - An explicit 307 redirect to B's per-cluster subdomain is only a FALLBACK: a host change drops host-only cookies, so it preserves the session ONLY if cookies are PARENT-DOMAIN-scoped or the session is an OIDC/JWT bearer. amoebius's Keycloak-OIDC/JWT sessions are portable, so it works -- but the transparent proxy avoids the concern entirely, so proxy is preferred.
      - Long-lived connections (WebSocket/SSE): the A-as-proxy forwards them to B, or sends a graceful close so the client reconnects and re-resolves; the in-cluster backend cutover uses Gateway-API HTTPRoute WEIGHTS (already the doctrine mechanism -- notes line 1 / release_lifecycle) for a gradual A->B shift.
      - UNPLANNED failover (A is DOWN, so no A-proxy exists): clients cached to A's dead IP get connection errors, retry, re-resolve (bounded by the low TTL), and rebind to B. Rebind is GUARANTEED (B holds the last-replicated session state; OIDC/JWT is portable) but NOT seamless -- brief client errors, and post-fork sessions re-auth.
      - ILLEGAL STATE FORECLOSED -- "a session that can't rebind to the new gateway." Model the migration as a typed GADT-indexed state machine (illegal_state_catalog §4.3) whose transitions are ORDERED and gated on OBSERVED edges (readiness-DAG discipline, no timers): stand-up-replica -> quiesce(A) -> drain/verify-caught-up -> promote(B) -> [A-ingress=PROXY + repoint DNS -> B] -> unfreeze(B) -> drain-monitor(A traffic -> 0) -> decommission(A-ingress).
